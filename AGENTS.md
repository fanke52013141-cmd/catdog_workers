# 呼噜噜 × 啊呜呜 AI 视频项目协作规则

本项目用于持续构建一个抖音虚拟 IP 短视频账号：主 IP 呼噜噜，副 IP 啊呜呜。账号的重点不是展示 AI 技术，而是用 AI 生产可记忆、可复用、可商业化的虚拟打工人内容。

## 核心原则

- 所有内容优先服务角色记忆点：呼噜噜是低电量猫系打工人，啊呜呜是吃货碎嘴狗系打工人。
- 不把视频做成单纯 AI 炫技。技术感可以存在，但用户记住的必须是角色、情绪和固定梗。
- 每次新增素材都要判断是否能复用：角色、表情、动作、场景、道具、台词、栏目、音效、提示词。
- 先建立最小可用资产库，再追求复杂世界观。
- 内容调性保持“可爱、真实、疲惫、好笑、治愈”，避免成功学、说教、营销号腔。

## 账号定位

一句话定位：

> 两个虚拟打工人女孩，用猫系低电量和狗系吃货碎嘴的反差，演出年轻人的上班、下班、续命、吐槽和自救。

用户看到账号时应快速理解：

- 呼噜噜：想努力但身体不允许，靠咖啡续命，发卡显示电量。
- 啊呜呜：嘴停不下来，饿了就关机，靠奶茶和零食重启，尾巴显示状态。
- 双人关系：互相嫌弃但离不开，是同病相怜的打工搭子。

## 标准工作流

1. 更新或补齐 IP 资产。
2. 从栏目库选择内容方向。
3. 写 15-35 秒短视频脚本。
4. 拆成分镜和 AI 生成提示词。
5. 生成视频/图片/音频素材。
6. 审核角色一致性、短视频节奏和商业风险。
7. 记录失败案例，反哺资产库、模板和提示词。

## 产物归档规则

- 角色设定和稳定参考放入 `references/`。
- 业务参数放入 `config/task.yaml`。
- 可复用模板放入 `templates/`。
- 项目规划和缺失项放入 `docs/`。
- 单条视频成片脚本、分镜、提示词放入 `outputs/`。
- 复盘记录放入 `logs/`。
- 失败案例放入 `bad_cases/bad_case_log.yaml`。

## 内容禁区

- 不让角色变成完美精致女神，必须保留“普通打工人”的真实感。
- 不让呼噜噜过度积极鸡血，她的冲劲必须短暂。
- 不让啊呜呜只剩吃货标签，她还要承担节奏、吐槽和行动力。
- 不做单纯办公室抱怨，结尾尽量有轻微治愈、荒诞反转或角色关系推进。
- 不为了热点牺牲人设稳定性。
